## Constraints

- Touch only files that are load-bearing for this failure. No drive-by refactors, no style "improvements".
- Match the existing code style of the file you are editing.
- Do NOT use `as any`, `@ts-ignore`, `@ts-expect-error`, `eslint-disable`, `#[allow(...)]`, `# noqa`, `# type: ignore` to silence an error **without** an inline `// justified: <reason>` comment AND a rule entry.
- Do NOT delete or weaken an existing failing test to reach green.
- Do NOT remove a guardrail (auth check, validation, permission gate, security middleware).
- If the fix requires changing security policy, permissions, or auth — STOP and return `status: blocked-needs-human`.

## Plan

1. **Reproduce** the failure by re-running the failing command in `<cwd>`.
   → verify: the same error message appears in the output.
2. **Confirm root cause** by reading the implicated code and naming the exact line(s).
   → verify: you can point to `file:line` and explain the mechanism in one sentence.
3. **Apply the minimal fix** in the load-bearing file(s).
   → verify: `git diff --stat` shows only the intended files changed.
4. **Run the relevant test suite** (or write a failing test first if none exists).
   → verify: test output shows the previously-failing case now passing.
5. **Run lint and typecheck** for the affected language (`tsc --noEmit`, `biome check`, `eslint`, `ruff check`, `cargo clippy`, `go vet`).
   → verify: exit code 0, no new warnings introduced.
6. **Persist a one-line rule** in the correct location (see decision tree below).
   → verify: the rule file exists and contains exactly one new operational rule.
7. **Update the harness index** if a new rule file was created.
   → verify: `AGENTS.md` (or `CLAUDE.md`) references the new rule file.

## Feedback — DO

- DO re-run the failing command before editing, to confirm reproduction.
- DO write a failing test first when no test covers the failure.
- DO keep the rule to ONE operational sentence.
- DO paste real verification output (not "tests passed" paraphrase).
- DO respect the project's `AGENTS.md` rules (e.g. marketplace sync, worktree hygiene).

## Feedback — DO NOT

- DO NOT conclude without running verification. "Looks right" is not evidence.
- DO NOT widen scope. If you notice unrelated dead code, mention it — don't delete it.
- DO NOT silence errors with type casts or linter disables (see Constraints).
- DO NOT batch multiple unrelated fixes into one run.
- DO NOT invent repo context you haven't read. If unsure, ask before editing.
- DO NOT update the global skill when the learning is project-specific (and vice-versa).
